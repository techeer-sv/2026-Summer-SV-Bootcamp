# 02-03. 컨벤션 합의 템플릿

📎 세션 슬라이드 15 (커밋 메시지 컨벤션)

기술적인 룰(보호 룰)은 끝났으니, 이제 **사람 간의 약속**을 정해요. 컨벤션이 없는 팀의 PR은 5명이 5가지 스타일을 써서 리뷰가 산만해집니다.

---

## 1. 컨벤션이란 무엇을 정하나요

부트캠프 4주에 합의해두면 좋은 4가지:

| 항목 | 예시 |
| --- | --- |
| **브랜치 네이밍** | `feat/#1-login-form` |
| **커밋 메시지** | `feat: 로그인 폼 추가` (Conventional Commits) |
| **PR 제목 & 본문** | PR 템플릿 5섹션 (요약/변경/테스트/스샷/Closes) |
| **이슈 제목 & 라벨** | 제목 prefix는 커밋과 동일, 라벨은 4~5개로 제한 |

이 4가지를 한 페이지로 정리한 게 `CONTRIBUTING.md`. 팀원이 첫날 한 번 읽고, 그다음부터는 막힐 때만 펴봐요.

---

## 2. `CONTRIBUTING.md` 1쪽 템플릿

레포 루트에 `CONTRIBUTING.md` 라는 파일을 만들어 아래를 복붙. 팀 회의 30분 안에 빈칸을 채우시면 됩니다.

````markdown
# Contributing — 팀 협업 가이드

> 이 문서는 우리 팀의 협업 약속이에요. 처음 합류했다면 한 번 정독,
> 작업 중 막히면 펴보세요.

## 1. 브랜치 네이밍

```
<type>/#<이슈번호>-<짧은-설명>
```

| Type | 의미 |
| --- | --- |
| `feat` | 새 기능 |
| `fix` | 버그 수정 |
| `docs` | 문서만 |
| `refactor` | 동작 그대로, 구조 개선 |
| `style` | 포맷·세미콜론 등 |
| `test` | 테스트 추가/수정 |
| `chore` | 빌드·설정·의존성 |

예시:
- `feat/#3-login-form`
- `fix/#7-csv-import-encoding`
- `docs/#12-api-readme`

## 2. 커밋 메시지

Conventional Commits 형식.

```
<type>: <한 줄 요약, 50자 이내>

[필요 시 본문]
```

- 한국어 / 영어 모두 OK — 팀 안에서 통일
- 한 커밋 = 한 논리적 변경 (작게 자주)

좋은 예:
- `feat: 로그인 폼 추가`
- `fix: 비밀번호 빈 값 제출 시 alert`
- `docs: README에 환경 변수 설정 추가`

나쁜 예:
- `update`
- `ㅋㅋ 버그`
- `feat: 로그인 + 회원가입 + 비밀번호 찾기` (덩어리 너무 큼)

## 3. PR 제목 & 본문

### 제목

커밋과 같은 컨벤션:
```
<type>: <요약>
```

### 본문 — 5섹션 템플릿

```markdown
## 요약
한두 줄로 무엇을 했는지

## 변경사항
- 항목 1
- 항목 2

## 테스트
- [ ] 무엇을 어떻게 확인했는지

## 스크린샷 (UI 변경 시)
(이미지 첨부)

## 관련 이슈
Closes #1
```

## 4. PR 리뷰 룰

- **머지에는 1명 이상의 Approve 필요**
- 본인 PR 본인 Approve 불가 (GitHub이 막음)
- 댓글이 달리면 작성자가 **Resolve conversation** 으로 닫음
- Squash 머지 통일
- 머지 후 가지 삭제 (자동 또는 수동)

## 5. 이슈 컨벤션

### 제목
```
<type>: <한 줄 요약>
```

### 라벨 (이번 팀에서 쓰는 것만)
- `feat` — 새 기능
- `fix` — 버그
- `docs` — 문서
- `discussion` — 논의 필요

### 본문 — 3섹션
```markdown
## TODO
- [ ] 항목 1

## 완료 기준
- 이 PR이 머지된다
- 이 화면이 동작한다

## 참고
- 관련 링크
```

## 6. 그 외 합의 (선택)

- 작업 시간대: 평일 9~22시 (그 외 시간은 답장 천천히)
- 데일리 스탠드업: 평일 10시 (Slack)
- 머지된 main을 매일 아침 본인 로컬에 pull
- 막히면 30분 안에 팀에 공유

## 7. 막힐 때

- [Git 사이클 가이드](./practice/) 참고
- 팀 Slack 채널 `#git-help`
- 멘토 호출

---

마지막 수정: 2026-MM-DD · 합의자: 팀원 전원
````

---

## 3. PR 본문 자동 채우기 — Pull Request Template

매 PR 만들 때마다 5섹션을 외워서 적기 귀찮죠. GitHub은 PR 본문을 자동으로 채워주는 템플릿 기능이 있어요.

### 설정 방법

레포 루트에 다음 경로로 파일 생성:

```
.github/PULL_REQUEST_TEMPLATE.md
```

내용:

```markdown
## 요약


## 변경사항
- 

## 테스트
- [ ] 

## 스크린샷 (UI 변경 시)


## 관련 이슈
Closes #
```

main에 머지된 직후부터, **새 PR을 만들 때 본문이 위 양식으로 자동으로 채워져 있습니다.** 팀원이 본문 양식을 매번 외울 필요가 없어져요.

> 💡 **Issue 템플릿도 가능합니다.** `.github/ISSUE_TEMPLATE/feature.md`, `.github/ISSUE_TEMPLATE/bug.md` 처럼 여러 개 만들 수 있어요. 부트캠프 4주에는 PR 템플릿 1개만 있어도 충분.

---

## 4. 컨벤션 합의 미팅 — 30분 안에

처음 모인 팀이 컨벤션 합의에 시간을 너무 쓰면 안 돼요. **이 자료의 권장값을 1차 안으로 두고**, 30분 안에 결정.

### 진행 흐름

1. **5분** — 팀원 각자 위 템플릿 정독
2. **10분** — 다른 의견이 있는 항목만 토론
   - 한국어 vs 영어
   - Required approvals 1 vs 2
   - 라벨 종류
3. **5분** — `CONTRIBUTING.md` 빈칸 채우기 (한 명이 PR로 올림)
4. **5분** — PR 리뷰 + 머지
5. **5분** — `.github/PULL_REQUEST_TEMPLATE.md` 작성 + 머지

이 미팅 자체가 **첫 PR 굴려보는 연습** 이에요. 일석이조.

---

## 5. 실제 동작 확인

위 두 파일이 main에 머지된 직후, 팀원 누구든 새 PR을 만들어보세요.

- 본문 입력칸에 5섹션 양식이 **자동으로 채워져 있는지** 확인
- 이게 보이면 팀 컨벤션이 본격 가동 중

---

## 🩺 막힐 때

<details>
<summary><b>PR 템플릿이 자동으로 안 보여요</b></summary>

- 파일 경로 정확히 확인: <code>.github/PULL_REQUEST_TEMPLATE.md</code> (소문자, 정확한 이름)
- main에 머지된 후에만 동작. 기존 PR은 영향 없음
- 브라우저 캐시: 시크릿 창에서 다시 시도

</details>

<details>
<summary><b>컨벤션 두고 팀원 의견이 갈려요</b></summary>

타협 안 되는 항목은 일단 본 자료의 권장값을 따르고, **1주일 후 회고에서 다시 토론**. 시작이 늦어지는 게 더 큰 문제예요.

</details>

<details>
<summary><b>CONTRIBUTING.md 가 너무 길어요</b></summary>

위 템플릿을 그대로 다 쓰지 않아도 됩니다. 정말 필요한 항목만. 단, <b>브랜치 네이밍 + 커밋 컨벤션 + 머지 룰</b> 셋은 빠뜨리지 마세요.

</details>
