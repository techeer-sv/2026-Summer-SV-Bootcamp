# 03-02. 안전한 되돌리기

📎 세션 슬라이드 24, 25 (RISK 2 · 3 — `revert` · `stash`)

세션에서 "사고가 났을 때 활용할 수 있는 손잡이 명령어들" 이라고 했던 그 명령들. 이 자료는 그 중 **안전한 두 가지 — `revert` · `stash`** 만 깊이 다룹니다.

> ⚠️ `reset --hard`, `push --force`, `rebase -i`, `cherry-pick`, `reflog`, `detached HEAD` 같은 위험 명령은 권장 동선에 등장하지 않아요. 이 챕터 끝의 **🚧 멘토/팀에 물어보기** 박스에만 라우팅 정보를 둡니다.

## 1. `git revert` — 안전한 되돌리기

세션 슬라이드 24의 그 명령. **기존 히스토리는 그대로 두고, "그 커밋을 취소하는 새 커밋"** 을 추가해요.

```
원래:        A ─ B ─ C ─ D                  (D 가 문제)
revert 후:   A ─ B ─ C ─ D ─ revert(D)      (D의 변경을 거꾸로 적용한 커밋)
```

### 언제

- 이미 push된 커밋의 동작을 되돌리고 싶을 때
- main에 들어간 커밋이 버그를 가져왔을 때
- 다른 팀원이 그 커밋 위에 작업 중일 가능성이 있을 때

### 사용법

```bash
# 1. 되돌릴 커밋의 SHA를 확인
$ git log --oneline
4f2c1ab (HEAD -> main) feat: 로그인 폼 추가  ← 이걸 되돌리고 싶음
a1b2c3d Initial commit

# 2. revert
$ git revert 4f2c1ab

# 3. 커밋 메시지가 자동으로 채워진 에디터가 뜸
# Revert "feat: 로그인 폼 추가"
# This reverts commit 4f2c1ab...
# (이대로 저장하고 종료)

# 4. push
$ git push
```

`git log` 를 다시 보면:

```bash
$ git log --oneline
b5e6f7g (HEAD -> main) Revert "feat: 로그인 폼 추가"
4f2c1ab feat: 로그인 폼 추가
a1b2c3d Initial commit
```

**원래 커밋은 그대로** + **반대 동작의 새 커밋**이 추가됐어요.

### 부트캠프에서의 표준 동선

1. main 에 실수로 들어간 커밋이 있다 → 팀에 공유
2. 새 브랜치 `fix/#N-revert-broken` 생성
3. `git revert <SHA>` → 커밋 → push → PR
4. PR 리뷰 → 머지

**main에 직접 revert를 push하지 마세요.** 보호 룰이 막을 거예요. PR을 거쳐서.

## 2. `git stash` — 잠깐 치워두기

세션 슬라이드 25의 그 명령. 작업 중인 변경을 **커밋하지 않고 임시 보관**.

### 언제

- 작업 중인데 급한 다른 일을 해야 할 때 (예: main에 핫픽스 필요)
- 잘못된 브랜치에 작업하고 있었을 때 (FAQ #3)
- pull 받으려는데 로컬 변경이 있어 거부될 때

### 시나리오 — 작업 중 핫픽스

```bash
# 1. 지금 feat/#1-login에서 작업 중. 아직 커밋 안 함
$ git status
On branch feat/#1-login
modified:   src/Login.tsx

# 2. 멘토가 "main의 결제 버그 30분 안에 고쳐주세요" 요청
$ git stash
Saved working directory and index state WIP on feat/#1-login: 4f2c1ab feat: 로그인 폼 추가

$ git status
On branch feat/#1-login
nothing to commit, working tree clean        ← 깨끗해짐

# 3. 핫픽스 작업
$ git switch main
$ git pull
$ git switch -c fix/#5-payment-bug
# ... 버그 수정 ...
$ git add .
$ git commit -m "fix: 결제 버튼 클릭 시 두 번 호출"
$ git push -u origin fix/#5-payment-bug
# PR → 머지

# 4. 원래 작업으로 복귀
$ git switch feat/#1-login
$ git stash pop
On branch feat/#1-login
modified:   src/Login.tsx                    ← 그대로 돌아옴
```

### 자주 쓰는 stash 명령

| 명령 | 의미 |
| --- | --- |
| `git stash` 또는 `git stash save` | 현재 변경을 보관 |
| `git stash push -m "메시지"` | 메시지와 함께 보관 (여러 stash 구분에 유용) |
| `git stash list` | 보관된 stash 목록 |
| `git stash pop` | 가장 최근 stash를 꺼내고 stash에서 삭제 |
| `git stash apply` | 꺼내기만 (stash 에 남겨둠 — 여러 번 적용 가능) |
| `git stash drop` | 가장 최근 stash 삭제 |
| `git stash clear` | 모든 stash 삭제 |

### `git stash list` 예시

```bash
$ git stash list
stash@{0}: WIP on feat/#1-login: 4f2c1ab feat: 로그인 폼 추가
stash@{1}: On main: WIP on README
```

특정 stash 만 적용:

```bash
$ git stash pop stash@{1}
```

### 추적 안 되는 파일도 같이

`.gitignore` 에 안 들어 있는 새 파일은 기본 stash에 안 포함돼요. 같이 보관하려면:

```bash
$ git stash --include-untracked
# 또는 짧게
$ git stash -u
```

## 3. revert vs stash — 한 줄 정리

| | revert | stash |
| --- | --- | --- |
| **무엇을** | 이미 commit된 변경 | 아직 commit 안 한 변경 |
| **결과** | 새 커밋이 추가됨 (히스토리 유지) | 임시 보관함에 들어감 |
| **언제** | "이 변경을 취소하고 싶다" | "잠깐 다른 일 해야 한다" |
| **위험도** | 안전 | 안전 |

## 4. 실습 — revert · stash 한 번씩

### revert 실습

```bash
# 가짜 잘못된 커밋 만들기
$ git switch -c oops/#test
$ echo "broken" >> README.md
$ git add README.md
$ git commit -m "feat: 일부러 깨뜨리는 변경"
$ git push -u origin oops/#test

# PR 머지까지 가지 마시고, 로컬에서 바로 revert 실습
$ git revert HEAD
# 에디터 뜨면 그대로 저장
$ git log --oneline
# Revert "feat: 일부러 깨뜨리는 변경"  ← 새 커밋
# feat: 일부러 깨뜨리는 변경              ← 원래 커밋도 남아 있음

# 정리
$ git switch main
$ git branch -D oops/#test
$ git push origin --delete oops/#test    # 원격도 정리
```

### stash 실습

```bash
$ git switch -c stash/#test
$ echo "WIP" >> README.md
$ git stash
$ git status        # 깨끗
$ git stash list
# stash@{0}: WIP on stash/#test: ...

$ git stash pop
$ git status        # README.md 변경 그대로

# 정리
$ git restore README.md
$ git switch main
$ git branch -D stash/#test
```

## 5. 🚧 멘토/팀에 물어보기 — 위험 명령이 필요해 보일 때

다음 상황이 보이면 **본인 손으로 해결하지 마시고** 멘토 또는 팀에 알리세요. 위험 명령은 잘못하면 팀 작업물을 영구히 날립니다.

| 상황 | 필요해 보이는 명령 (이 자료에서 안 다룸) | 이유 |
| --- | --- | --- |
| 마지막 커밋을 통째로 없애고 싶음 | `git reset --hard HEAD~1` | 로컬 변경이 영구 손실, 팀과 동기화 깨짐 |
| main의 히스토리를 깔끔하게 재정리하고 싶음 | `git rebase -i` (interactive rebase) | 히스토리 재작성 = 팀원 로컬과 충돌 |
| push 후에 강제로 덮어쓰고 싶음 | `git push --force` | 팀원 작업 날림 — **세션 슬라이드 24에서 강조한 그 명령** |
| 다른 브랜치의 한 커밋만 가져오고 싶음 | `git cherry-pick <SHA>` | 의도 명확하지 않으면 히스토리 꼬임 |
| 이전 상태로 시간 여행하고 싶음 | `git reflog`, `detached HEAD` | 정상 동선에서 벗어나, 초보자에게 패닉 유발 |
| 커밋 분할/합치기 | `git rebase -i squash/edit` | 히스토리 재작성 |

**위 명령들이 정말 필요하다면 멘토가 함께 진행해줄 거예요.** 혼자 인터넷에서 명령어 복사해 실행하지 마세요.

## 🩺 막힐 때

<details>
<summary><b>revert 도중 충돌이 났어요</b></summary>

revert도 일종의 머지라 충돌이 가능해요. <a href="../02-팀과-같이-쓰기/05-conflict-해결.md">02-05 Conflict 해결</a> 의 동선과 동일하게 마커 편집 → add → <code>git revert --continue</code>.

중단하려면:
```bash
$ git revert --abort
```

</details>

<details>
<summary><b>stash pop 도중 충돌이 났어요</b></summary>

stash 안의 변경과 현재 브랜치의 변경이 같은 줄에 있을 때. VSCode로 마커 해결 → add → commit. stash는 자동으로 비워집니다.

</details>

<details>
<summary><b>stash에 뭐가 있었는지 까먹었어요</b></summary>

```bash
$ git stash list                    # 목록
$ git stash show -p stash@{0}       # 가장 최근 stash의 변경 내용 자세히
```

</details>

<details>
<summary><b>revert를 했는데 다시 그 변경을 적용하고 싶어요</b></summary>

revert의 revert: 그냥 다시 한 번 `git revert <revert-commit-SHA>` 하시면 됩니다. revert도 일반 커밋이라 revert 가능.

</details>
