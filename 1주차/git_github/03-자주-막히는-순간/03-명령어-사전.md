## 1. 상태 확인 4종

작업 중 "지금 내 위치 어디지?" 가 막힐 때, 이 4개만 알면 충분해요.

| 명령 | 무엇을 보여주나 |
| --- | --- |
| `git status` | 작업 디렉토리·스테이지 변경 + 원격과의 차이를 한 화면에 |
| `git branch` | 로컬 브랜치 목록 + `*` 로 현재 브랜치 표시 |
| `git log --oneline` | 커밋 히스토리 한 줄씩 (가장 가독성 좋음) |
| `git remote -v` | 연결된 원격 저장소 URL |

### `git status` 한 번에 읽기

```bash
$ git status
On branch feat/#1-login          ← (a) 현재 브랜치
Your branch is up to date with 'origin/feat/#1-login'.  ← (b) 원격과 동기화 상태

Changes to be committed:          ← (c) 스테이지에 올라간 변경 (다음 commit 대상)
  (use "git restore --staged <file>..." to unstage)
        modified:   src/Login.tsx

Changes not staged for commit:    ← (d) 작업 디렉토리에만 있는 변경
  (use "git add <file>..." to update what will be committed)
        modified:   README.md

Untracked files:                  ← (e) Git이 처음 보는 파일
  (use "git add <file>..." to include in what will be committed)
        notes.txt
```

| 위치 | 영역 | 이 상태에서 git commit 하면 |
| --- | --- | --- |
| (c) | Staging | ✅ commit에 포함됨 |
| (d) | Working Directory | ❌ 무시됨 (먼저 `add` 해야) |
| (e) | (추적 안 됨) | ❌ 무시됨 |

> 💡 4단계 영역(Working · Staging · Local · Remote)을 다시 보고 싶으면 [01-04 Branch와 커밋](../01-한사이클-혼자-돌려보기/04-branch-와-커밋.md#4-commit--사진-한-컷씩-찍기) 의 다이어그램으로.

### `git log --oneline` — 자주 쓰는 옵션

```bash
$ git log --oneline                        # 한 줄씩
$ git log --oneline --graph --all          # 모든 브랜치를 그래프로 (가장 예쁨)
$ git log --oneline -n 5                   # 최근 5개만
```

## 2. diff — 변경 확인

"이 파일 뭐 바꿨더라" 가 막힐 때.

| 명령 | 무엇 |
| --- | --- |
| `git diff` | 아직 `add` 안 한 변경 (Working ↔ Staging) |
| `git diff --staged` | `add` 했지만 commit 안 한 변경 (Staging ↔ 마지막 커밋) |
| `git diff HEAD` | 마지막 커밋 이후 모든 변경 (Working + Staging 합쳐서) |
| `git show <SHA>` | 특정 커밋 1개의 내용 (메시지 + diff) |
| `git diff <SHA1> <SHA2>` | 두 커밋 사이의 변경 |
| `git diff origin/main` | 원격 main 과 내 현재 브랜치의 차이 |

### 실전 예시

```bash
# 방금 수정한 게 무엇이었지?
$ git diff
diff --git a/src/Login.tsx b/src/Login.tsx
@@ -10,3 +10,5 @@
   const [email, setEmail] = useState('');
+  const [password, setPassword] = useState('');
+  ...

# 어제 푸시한 PR 의 변경이 main 과 얼마나 차이 나지?
$ git fetch origin
$ git diff origin/main
```

> 💡 VSCode 에서는 좌측 Source Control 패널에서 파일을 클릭하면 같은 diff가 시각적으로 보입니다. 터미널 diff 가 어색할 때 활용.

## 3. stage 에서 빼기 / 작업 되돌리기

세 가지 "되돌리기" 가 헷갈려요. 케이스별로.

### 케이스 A. `git add` 했는데 이번 commit 에서 빼고 싶다

```bash
$ git restore --staged <파일>
# 또는 옛 명령: git reset HEAD <파일>
```

파일 내용은 **그대로**, "Staging 에서만" 빠집니다. 즉 Working Directory 의 변경은 살아 있음.

### 케이스 B. 작업한 내용을 통째로 버리고 싶다 (commit 전)

```bash
$ git restore <파일>
# 또는 옛 명령: git checkout -- <파일>
```

⚠️ **이 변경은 영구히 사라집니다.** Git 이 기록을 보관하지 않는 상태(=add 도 안 된 변경)였기 때문에 복구가 어려워요. 정말 버려도 되는지 한 번 더 확인하세요.

### 케이스 C. 이미 추적 중인 파일을 추적에서만 빼고 로컬엔 남기고 싶다

`.env` 같은 파일을 실수로 add 한 직후.

```bash
$ git rm --cached <파일>
```

파일은 디스크에 남고, Git 추적 대상에서만 빠집니다. 보통 `.gitignore` 보강과 함께 씁니다. 자세한 동선은 [01-02 .gitignore와 .env](../01-한사이클-혼자-돌려보기/02-gitignore-와-env.md#4-함정--이미-추적-중인-파일은-gitignore-가-못-막아요).

### 한눈에 비교

| 케이스 | 명령 | 파일 내용 | Git 추적 |
| --- | --- | --- | --- |
| A. stage 에서 빼기 | `git restore --staged <f>` | 그대로 | 그대로 추적 |
| B. 작업 버리기 | `git restore <f>` | **마지막 commit 으로 덮어쓰기** | 그대로 추적 |
| C. 추적에서 빼기 | `git rm --cached <f>` | 그대로 | 추적 제외 |

## 4. 옛 명령어 → 새 명령어 매핑

`git checkout` 이 워낙 여러 일을 해서 헷갈리게 만들어, 2019년 (Git 2.23) 부터 `switch` (브랜치 이동) 와 `restore` (파일 복원) 가 새로 생겼어요. 둘 다 동작합니다.

| 옛 명령 (`checkout`) | 새 명령 | 의미 |
| --- | --- | --- |
| `git checkout main` | `git switch main` | 브랜치 이동 |
| `git checkout -b feat/#1` | `git switch -c feat/#1` | 새 브랜치 만들면서 이동 |
| `git checkout -- README.md` | `git restore README.md` | 파일을 마지막 커밋 상태로 복원 |
| `git checkout HEAD -- README.md` | `git restore --source=HEAD README.md` | 특정 커밋의 파일로 복원 |

> 💡 이 자료는 **새 명령(`switch`/`restore`)** 으로 통일합니다. 인터넷의 옛 답변을 보고 `checkout` 을 그대로 따라 쳐도 동작은 하지만, 의미가 한눈에 안 들어와서 초보자에게는 헷갈려요.

## 5. log 검색

커밋이 50~100개 쌓이면 원하는 커밋을 찾기 어려워져요.

| 의도 | 명령 |
| --- | --- |
| 특정 메시지가 포함된 커밋 | `git log --grep="로그인"` |
| 특정 사람이 만든 커밋 | `git log --author="Leo"` |
| 특정 날짜 이후 커밋 | `git log --since="2026-06-01"` |
| 특정 파일을 건드린 커밋 | `git log -- src/Login.tsx` |
| 특정 파일의 각 줄을 누가 마지막으로 바꿨는지 | `git blame src/Login.tsx` |
| 한 줄에 보기 + 그래프 | `git log --oneline --graph --all` |

### 실전 — 어제 만든 커밋만 보기

```bash
$ git log --oneline --since="1 day ago"
4f2c1ab feat: 로그인 폼 추가
b5e6f7g docs: README 자기소개
```

## 6. 두 지점 사이 비교

"내 브랜치와 main 사이엔 어떤 변경이 있지?" 같은 질문.

| 비교 대상 | 명령 |
| --- | --- |
| 두 커밋 사이의 변경 (diff) | `git diff <SHA1> <SHA2>` |
| 두 커밋 사이의 커밋 목록 | `git log <SHA1>..<SHA2> --oneline` |
| 내 브랜치가 main 보다 앞선 커밋들 | `git log main..HEAD --oneline` |
| main 이 내 브랜치보다 앞선 커밋들 | `git log HEAD..main --oneline` |
| 원격 main 과 내 현재 브랜치 차이 | `git fetch origin && git diff origin/main` |

### 실전 — PR 만들기 전 변경 미리 확인

```bash
$ git fetch origin
$ git log origin/main..HEAD --oneline       # 이 PR 에 들어갈 커밋들
$ git diff origin/main                       # 변경 전체 diff
```

## 7. 원격과의 상태 차이 읽기

`git status` 가 알려주는 한 줄짜리 메시지의 의미.

| 메시지 | 의미 | 다음 행동 |
| --- | --- | --- |
| `Your branch is up to date with 'origin/...'.` | 동기화 됨 | 작업 계속 |
| `Your branch is ahead of 'origin/...' by N commits.` | 로컬이 더 앞섬 (push 안 함) | `git push` |
| `Your branch is behind 'origin/...' by N commits, and can be fast-forwarded.` | 원격이 더 앞섬 (다른 사람이 push 함) | `git pull` |
| `Your branch and 'origin/...' have diverged, and have N and M different commits each.` | 양쪽이 다르게 갈라짐 | `git pull --rebase` 또는 `git pull` (충돌 가능) |

### fetch vs pull — 한눈에

| 명령 | 무엇 |
| --- | --- |
| `git fetch origin` | 원격 변경을 **받기만** 함. 내 작업 디렉토리·브랜치엔 영향 없음 |
| `git pull origin` | `fetch + merge` 를 한 번에. 원격 변경을 내 브랜치에 자동 병합 (충돌 가능) |
| `git pull --rebase` | `fetch + rebase`. 머지 커밋 없이 일직선 |

> 💡 "원격에 뭐가 있는지 보고만 싶다" 면 `git fetch`, "받아서 바로 내 작업에 합치고 싶다" 면 `git pull`. 충돌이 걱정될 땐 fetch 로 먼저 확인하는 게 안전합니다.

## 8. stash 살펴보기

stash 의 기본 동작은 [03-02 안전한 되돌리기 — 2. stash](./02-안전한-되돌리기.md#2-git-stash--잠깐-치워두기) 를 보시고, 여기는 **"어디 갔지" 가 막힐 때** 의 도구 모음.

| 의도 | 명령 |
| --- | --- |
| 보관된 stash 목록 | `git stash list` |
| 가장 최근 stash 의 변경 내용 자세히 | `git stash show -p` |
| 특정 stash 의 변경 내용 | `git stash show -p stash@{1}` |
| 가장 최근 stash 꺼내기 + 삭제 | `git stash pop` |
| 가장 최근 stash 꺼내기 (보관 유지) | `git stash apply` |
| 특정 stash 꺼내기 | `git stash pop stash@{1}` |
| 가장 최근 stash 삭제 | `git stash drop` |
| 모든 stash 삭제 | `git stash clear` |
| 메시지 붙여서 stash | `git stash push -m "WIP 로그인"` |
| 추적 안 되는 파일도 함께 stash | `git stash -u` |

### 실전 — 며칠 전 stash 가 있나 확인

```bash
$ git stash list
stash@{0}: WIP on feat/#1-login: 4f2c1ab feat: 로그인 폼 추가
stash@{1}: On main: WIP README

$ git stash show -p stash@{1}
# (변경 내용 diff)

$ git stash pop stash@{1}    # 마음에 들면 꺼내기
```

## 9. 브랜치 청소

머지된 브랜치가 로컬·원격에 죽은 채로 남아 있으면 관리가 어려워요.

| 의도 | 명령 |
| --- | --- |
| 로컬 브랜치 목록 | `git branch` |
| 원격까지 포함 | `git branch -a` |
| **머지된 로컬 브랜치만** 보기 | `git branch --merged` |
| 머지된 로컬 브랜치 삭제 (안전) | `git branch -d <이름>` |
| 강제 삭제 (안 머지된 것도) | `git branch -D <이름>` ⚠️ |
| 원격 브랜치 삭제 | `git push origin --delete <이름>` |
| **원격에서 사라진 브랜치 참조 정리** | `git fetch --prune` |
| 머지된 로컬 브랜치 한 번에 정리 (main/develop 제외) | `git branch --merged \| grep -vE '(^\*\|main\|develop)' \| xargs -n 1 git branch -d` |

### 머지 후 표준 청소 동선

GitHub PR 머지 직후, 로컬에서:

```bash
$ git switch main
$ git pull                          # 머지된 변경을 로컬 main 에 반영
$ git branch -d feat/#1-login       # 로컬 브랜치 삭제
$ git fetch --prune                 # 원격에서 사라진 가지 참조 정리
```

이 4줄을 머지 직후 항상 돌리는 습관을 들이세요.

> 💡 GitHub 레포 설정 → **Settings → General → Pull Requests → Automatically delete head branches** 를 켜두면 원격은 자동 정리됩니다. ([02-02](../02-팀과-같이-쓰기/02-보호-룰-3개.md#9-자동-머지-정리-옵션-보너스) 참고)

## 10. (참고) Git 공식이 알려주는 "이런 의도엔 이 명령"

명령 자체의 매뉴얼을 보고 싶을 때는 터미널에서:

```bash
$ git help <명령>          # 풀 매뉴얼 (q 로 종료)
$ git <명령> --help        # 동일
$ git <명령> -h            # 짧은 옵션 요약
```

예: `git help switch`, `git stash --help`, `git log -h`.

오프라인에서도 동작해서 인터넷이 끊겼을 때 유용합니다.
