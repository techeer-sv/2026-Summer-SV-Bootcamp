# 03-05. 에러 메시지 사전 — 메시지 그대로 검색

> 터미널에 나오는 빨간/노란 메시지를 그대로 검색해서 펴보는 사전이에요.
> 각 항목은 **메시지 → 원인 → 처방 → 예방** 4단.
> 메시지가 영어라 무서워 보여도, 거의 다 "**이거 해보세요**" 라고 알려주고 있는 문장이에요.

📎 세션 슬라이드 14, 22~25 + [03-01 FAQ](./01-faq.md) 와 짝꿍

---

## 0. 어떻게 쓰나요

1. 터미널 메시지의 **첫 줄 (보통 `error:` `fatal:` `hint:` 로 시작)** 을 그대로 복사
2. 이 챕터에서 `Cmd+F` / `Ctrl+F` 로 키워드 검색
3. 안 보이면 → [03-01 FAQ](./01-faq.md) → [03-04 시나리오별 복구](./04-시나리오별-복구.md) → 멘토

> 💡 **에러 메시지의 첫 줄을 복사해서 검색** 하는 습관을 들이세요. Git 의 거의 모든 에러는 **이미 다른 누군가가 만났던** 것이에요. 한국어 번역이 어색하면 영어 원문으로 검색하면 답이 훨씬 많이 나옵니다.

## 1. `fatal: not a git repository`

```
fatal: not a git repository (or any of the parent directories): .git
```

### 원인

지금 있는 폴더(와 상위 폴더 모두) 가 Git 저장소가 아니에요. `.git` 폴더가 없는 곳에서 git 명령을 친 상태.

### 처방

```bash
# 1. 어디 있는지 확인
$ pwd
/Users/me/Downloads          ← 😱 작업 폴더 아님

# 2. 올바른 폴더로 이동
$ cd ~/work/git-practice-2026
$ git status
On branch main               ← 정상
```

또는, **이 폴더를 새 Git 저장소로 만들고 싶다면** (드물지만):

```bash
$ git init
```

### 예방

- 터미널 프롬프트에 현재 브랜치를 표시하는 도구를 깔아두면 한눈에 보입니다 (예: oh-my-zsh, Starship)
- VSCode 의 터미널은 자동으로 작업 폴더로 진입해서 이 에러가 거의 안 나요

## 2. `nothing to commit, working tree clean`

```
On branch main
Your branch is up to date with 'origin/main'.

nothing to commit, working tree clean
```

### 원인

**에러 아닙니다.** 변경할 게 없다는 정상 메시지. 깨끗한 상태.

### 처방

축하합니다 🎉 — 다음 작업을 진행하시면 됩니다.

### 헷갈리는 케이스

"분명히 README 를 수정했는데 왜 clean 이라고 나오지?" — 다음을 확인:

- **다른 폴더의 같은 이름 파일을 수정**한 건 아닌지 (`pwd` 확인)
- 파일 저장을 안 한 건 아닌지 (VSCode 의 점 표시 `●` 가 파일명 옆에 있으면 미저장)
- `.gitignore` 에 들어 있어서 Git 이 무시하는 파일이 아닌지 (`git check-ignore <파일>` 로 확인)

## 3. `Your branch is behind 'origin/...'`

```
Your branch is behind 'origin/main' by 3 commits, and can be fast-forwarded.
  (use "git pull" to update your local branch)
```

### 원인

**원격이 내 로컬보다 앞서 있어요.** 다른 팀원이 push 한 변경이 쌓인 상태.

### 처방

메시지가 친절히 알려줍니다.

```bash
$ git pull
```

`fast-forward` 라고 적혀 있으면 **충돌 없이 자동 병합**. 안전합니다.

### 예방

- 작업 시작 전 항상 `git pull` 습관
- 자세한 동선: [03-04 시나리오 5](./04-시나리오별-복구.md#-시나리오-5--작업-중인데-main-이-갱신됐어요)

## 4. `Your branch is ahead of 'origin/...'`

```
Your branch is ahead of 'origin/main' by 2 commits.
  (use "git push" to publish your local commits)
```

### 원인

**내 로컬에 commit 은 했는데 push 를 안 했어요.** 정상 상태.

### 처방

```bash
$ git push
```

### 예방

- commit 후 push 까지 짝꿍처럼. VSCode Source Control 패널은 commit 옆에 push 버튼이 함께 보입니다.

## 5. `Your branch and 'origin/...' have diverged`

```
Your branch and 'origin/main' have diverged,
and have 2 and 3 different commits each, respectively.
  (use "git pull" if you want to integrate the remote changes)
```

### 원인

**양쪽이 다르게 갈라졌어요.** 내가 로컬에서 2 커밋 만드는 동안, 원격에서도 다른 3 커밋이 추가된 상태.

### 처방

```bash
$ git pull --rebase     # 원격 변경을 가져온 뒤 내 커밋을 그 위에 다시 쌓기 (일직선)
# 또는
$ git pull              # 머지 커밋 한 줄을 추가하면서 합치기
```

충돌이 같이 나면 → [02-05 Conflict 해결](../02-팀과-같이-쓰기/05-conflict-해결.md).

### 예방

- 자주 pull
- 같은 브랜치를 둘이 동시에 만지지 않기

## 6. `Your local changes ... would be overwritten by merge`

```
error: Your local changes to the following files would be overwritten by merge:
        src/Login.tsx
Please commit your changes or stash them before you merge.
Aborting
```

### 원인

`git pull` 또는 `git merge` 를 하려는데, **아직 commit 안 한 작업 중인 변경이 머지에 덮어쓰일 위험**이 있어요. Git 이 안전을 위해 거부합니다.

### 처방

세 가지 길:

```bash
# 옵션 A. 작업이 어느 정도 됐다면 — commit 후 pull
$ git add src/Login.tsx
$ git commit -m "WIP: 로그인 폼 진행 중"
$ git pull

# 옵션 B. 잠깐 치워두고 pull 후 다시 꺼내기
$ git stash
$ git pull
$ git stash pop

# 옵션 C. 작업한 게 별 가치 없으면 버리고 pull
$ git restore src/Login.tsx    # 변경 영구 삭제 ⚠️
$ git pull
```

가장 자주 쓰는 건 **옵션 B (stash)**. 자세한 동선: [03-02 stash](./02-안전한-되돌리기.md#2-git-stash--잠깐-치워두기).

### 예방

- pull 전에 항상 `git status` 로 작업 디렉토리가 깨끗한지 확인

## 7. `fatal: refusing to merge unrelated histories`

```
fatal: refusing to merge unrelated histories
```

### 원인

**서로 공통 조상이 없는 두 히스토리를 합치려는 시도.** 보통:

- 빈 폴더에서 `git init` 한 뒤, GitHub 에서 README 가 있는 레포를 추가로 만들었을 때
- 두 개의 독립된 Git 저장소를 하나로 합치려 할 때

> 📎 Git 2.9 부터 안전을 위해 기본 거부됩니다. 우연히 잘못된 두 프로젝트를 합치는 사고 방지가 목적.

### 처방

정말 합치는 게 맞다면 (보통 부트캠프 시작 시):

```bash
$ git pull origin main --allow-unrelated-histories
```

하지만 부트캠프 4주에서는 거의 다음이 정답:

```bash
# 1. 빈 폴더의 git init 을 취소
$ rm -rf .git

# 2. GitHub 레포를 clone (이쪽이 단순)
$ cd ..
$ git clone https://github.com/내-username/내-레포.git
```

### 예방

- 새 프로젝트는 **GitHub 에서 먼저 만들고 clone** (이 자료의 표준 동선)
- `git init` 은 이미 코드가 있고 그걸 Git 으로 관리 시작할 때만 ([01-01 clone vs init](../01-한사이클-혼자-돌려보기/01-레포-만들기.md#4-참고-clone-vs-init--언제-무엇을-쓰나요))

## 8. `failed to push some refs ... (non-fast-forward)` / `(fetch first)`

```
 ! [rejected]        feat/login -> feat/login (fetch first)
error: failed to push some refs to 'https://github.com/...'
hint: Updates were rejected because the remote contains work that you do
hint: not have locally.
hint: Integrate the remote changes (e.g. 'git pull ...') before pushing again.
```

### 원인

**원격의 같은 브랜치에 내 로컬보다 새로운 커밋이 있어요.** 보통:

- 다른 사람이 같은 브랜치에 push 했거나
- GitHub 웹에서 직접 파일을 수정했거나
- 직전에 PR 페이지에서 Suggestion 적용이 자동 커밋을 추가했거나

### 처방

```bash
$ git pull --rebase     # 원격 받고 내 커밋을 위에 다시 쌓기
$ git push              # 다시 시도
```

충돌이 같이 나면 [02-05 Conflict 해결](../02-팀과-같이-쓰기/05-conflict-해결.md).

자세한 건 [03-01 FAQ #1](./01-faq.md#-1--rejected--fetch-first).

### 예방

- push 전에 한 번 `git fetch` 로 원격 상태 확인하는 습관
- 같은 브랜치에 두 명이 동시 작업하지 않기

## 9. `error: src refspec ... does not match any`

```
error: src refspec feat/#1-login does not match any
error: failed to push some refs to 'https://github.com/...'
```

### 원인

push 하려는 **브랜치가 로컬에 존재하지 않거나, 이름을 잘못 쳤어요.**

### 처방

```bash
# 1. 실제 로컬 브랜치 확인
$ git branch
* feat/#1-login-form         ← 실제 이름
  main

# 2. 정확한 이름으로 다시
$ git push -u origin feat/#1-login-form
```

흔한 실수:
- `feat/#1-login` 대신 `feat/1-login` (`#` 누락)
- 한국어/영어 자판 차이로 보이지 않는 문자
- 옛 이름이 캐시에 남아 있음

### 예방

- 새 브랜치 만들 때 `git switch -c <이름>` 으로 만들면 이름이 그대로 push 됨
- VSCode 좌측 하단의 브랜치 이름을 그대로 복사해 쓰세요

## 10. `fatal: A branch named ... already exists`

```
fatal: A branch named 'feat/#1-login' already exists.
```

### 원인

이미 같은 이름의 브랜치가 로컬에 존재.

### 처방

```bash
# 옵션 A. 기존 브랜치로 이동만 하고 싶었다면
$ git switch feat/#1-login

# 옵션 B. 옛 브랜치를 버리고 새로 만들고 싶다면 (커밋 잃을 수 있음, 확인 후)
$ git branch -D feat/#1-login    # 강제 삭제 ⚠️
$ git switch -c feat/#1-login

# 옵션 C. 새 이름으로 만들기
$ git switch -c feat/#1-login-v2
```

### 예방

- `git branch` 로 먼저 확인 후 새 이름 짓기
- 머지된 가지를 바로바로 삭제 ([03-03 #9 브랜치 청소](./03-명령어-사전.md#9-브랜치-청소))

## 11. `error: pathspec '...' did not match any file(s) known to git`

```
error: pathspec 'README.md' did not match any file(s) known to git
```

### 원인

Git 이 그 이름의 파일/브랜치를 모릅니다. 보통:

- 파일 이름 오타 (`Readme.md` vs `README.md` 대소문자 차이 등)
- 다른 폴더에 있는 파일 (경로 누락)
- 새로 만든 파일인데 아직 `git add` 안 됨

### 처방

```bash
# 1. 실제 이름·경로 확인
$ ls
$ git status

# 2. 정확한 경로로 다시
$ git add docs/README.md
```

### 예방

- 탭 자동완성을 활용하세요. 셸이 정확한 이름을 채워줍니다
- macOS 의 기본 파일시스템은 대소문자 구분 안 함이지만 Git 은 구분합니다. 새 파일 만들 때 컨벤션 통일

## 12. `Support for password authentication was removed on August 13, 2021.`

```
remote: Support for password authentication was removed on August 13, 2021.
remote: Please see https://docs.github.com/get-started/getting-started-with-git/about-remote-repositories#cloning-with-https-urls for information on currently recommended modes of authentication.
fatal: Authentication failed for 'https://github.com/...'
```

### 원인

push 또는 clone 인증 시 **비밀번호 칸에 GitHub 비밀번호를 입력**했어요. GitHub 은 2021-08 부터 비밀번호 인증을 막았습니다.

### 처방

**PAT (Personal Access Token) 을 발급해서 비밀번호 칸에 붙여넣기.**

1. [github.com/settings/tokens](https://github.com/settings/tokens) → Generate new token (classic)
2. `repo` scope 체크, 만료 90일
3. 발급된 토큰을 **즉시 안전한 곳에 저장** (한 번만 보임)
4. 다음 push/clone 시 비밀번호 칸에 PAT 붙여넣기

자세한 동선: [00-04 인증](../00-환경세팅/04-인증.md).

### 예방

- 환경세팅 단계에서 한 번 발급해두면 부트캠프 끝까지 무탈
- VSCode 의 GitHub 자동 OAuth 가 가장 쉬워요 ([00-04](../00-환경세팅/04-인증.md))

## 13. `Permission ... denied` / 403 Forbidden

```
remote: Permission to user/repo.git denied to your-username.
fatal: unable to access 'https://github.com/...': The requested URL returned error: 403
```

### 원인

- **collaborator 권한 부족** (또는 Read 만 있음)
- 또는 **자격증명 캐시에 잘못된 토큰**이 끼어 있음

### 처방

#### 권한 부족이면

팀장에게 collaborator 추가 + **Write** 권한 요청. 본인이 정말 그 레포에 권한이 있는지 GitHub 페이지에서 확인 (브라우저에서 레포에 접근 가능한지).

#### 옛 자격증명이 캐시에 끼어 있으면

**macOS:**
```
Keychain Access 앱 → "github.com" 검색 → 항목 삭제 → 다시 push
```

**Windows:**
```
제어판 → 사용자 계정 → 자격 증명 관리자 → Windows 자격 증명
→ "git:https://github.com" 항목 제거 → 다시 push
```

### 예방

- PAT 만료일을 캘린더에 표시
- 회사·학교 노트북을 바꾸면 인증 다시 한 번

## 14. `SSL certificate problem`

```
fatal: unable to access 'https://github.com/...': SSL certificate problem: unable to get local issuer certificate
```

### 원인

**회사망/학교망의 프록시 또는 SSL 검사 장비** 가 GitHub 인증서를 가로채면서 발생. 거의 항상 회사 네트워크 문제.

### 처방

**1순위. IT 부서에 사내 인증서 설치 요청.** 회사 노트북은 사내 인증서를 시스템에 추가하는 절차가 보통 있어요.

**2순위. 핫스팟으로 잠깐 우회** — 핵심 작업만 빠르게.

**❌ 절대 하지 말 것:**
```bash
$ git config --global http.sslVerify false    # 보안 무력화. 절대 ❌
```

검색하면 자주 나오는 답변이지만, 회사 보안 정책 위반 + MITM 공격에 노출됩니다. 멘토와 IT 부서로 가세요.

### 예방

- 첫날 환경세팅 시 회사망에서 한 번 테스트
- 막히면 즉시 IT 부서 → 멘토 동시 호출

## 15. `warning: LF will be replaced by CRLF` (경고)

```
warning: in the working copy of 'README.md', LF will be replaced by CRLF the next time Git touches it
```

### 원인

**Windows 만의 줄바꿈 차이 경고.** 에러 아닙니다.

- Unix 계열(macOS/Linux) — 줄바꿈을 `LF` (`\n`)
- Windows — 줄바꿈을 `CRLF` (`\r\n`)

Git for Windows 가 기본 설정으로 **체크아웃 시 CRLF 로, commit 시 LF 로** 자동 변환합니다 (안전한 기본값).

### 처방

대부분 무시해도 됩니다. 동작에 영향 없음. 경고가 너무 시끄러우면:

```bash
# Windows
$ git config --global core.autocrlf true       # 기본값. 그대로 두면 됨

# macOS/Linux
$ git config --global core.autocrlf input      # commit 시에만 LF 변환
```

### 예방

- 팀원의 OS 가 섞여 있으면 레포 루트에 `.gitattributes` 추가:
  ```
  * text=auto eol=lf
  ```
  이렇게 하면 OS 와 무관하게 일관된 줄바꿈으로 통일됩니다.

## 16. `GH006: Protected branch update failed`

```
remote: error: GH006: Protected branch update failed for refs/heads/main.
remote: error: At least 1 approving review is required by reviewers with write access.
To https://github.com/.../...
 ! [remote rejected] main -> main (protected branch hook declined)
```

### 원인

**main 에 직접 push 하려 했는데 보호 룰이 막았어요.** 정상 동작입니다 — 보호 룰이 잘 걸려 있다는 증거.

### 처방

PR 동선으로 가세요.

```bash
# 1. 새 브랜치로 옮기기 (직전 커밋도 같이)
$ git switch -c fix/#N-my-change
$ git push -u origin fix/#N-my-change

# 2. 로컬 main 의 잘못된 커밋 풀어내기
$ git switch main
$ git reset --soft HEAD~1   # 마지막 커밋만 풀기 (변경은 유지 → 새 브랜치에 이미 있음)
$ git restore --staged .
$ git restore .

# 3. GitHub 에서 PR 만들기
```

### 예방

- 작업 시작 시 항상 `git branch` 로 현재 브랜치 확인
- main 에 직접 push 가 거부되는 게 정상. 이게 [02-02 보호 룰](../02-팀과-같이-쓰기/02-보호-룰-3개.md) 의 목적

## 17. `cannot lock ref` / `Unable to create '.git/index.lock': File exists`

```
fatal: Unable to create '/path/to/.git/index.lock': File exists.

Another git process seems to be running in this repository, e.g.
an editor opened by 'git commit'. Please make sure all processes
are terminated then try again.
```

### 원인

다른 Git 작업이 중간에 끊겼을 때 남은 잠금 파일이 그대로 있어요. 또는 VSCode 의 Git 확장이 백그라운드에서 동시에 명령을 실행 중일 수 있음.

### 처방

```bash
# 1. 다른 Git 프로세스가 정말로 안 돌고 있는지 확인
$ ps aux | grep git

# 2. 안 돌고 있으면 잠금 파일 제거
$ rm .git/index.lock

# 3. 다시 명령 시도
$ git status
```

> ⚠️ 다른 Git 프로세스가 정말 돌고 있는데 잠금 파일을 지우면 Git 의 인덱스가 깨질 수 있어요. **반드시 ps 확인 후** 지우세요.

### 예방

- VSCode 의 Source Control 패널과 터미널을 동시에 너무 빠르게 조작하지 않기
- `git commit` 시 에디터가 열린 채로 다른 터미널에서 commit 하지 않기
